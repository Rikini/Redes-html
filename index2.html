Fase 1 del Proyecto

Generación de Escenarios

Se creo una máquina virtual pfSense, Debian y Mint en una de las dos redes/vlans que se manejaron en el proyecto. Se les asignaron IPs disponibles a todos los equipos para comenzar a enlazarlos a internet. Para esto se configuraron dos adaptadores de red en pfSense uno para la red lan en la y el segundo para wan y tener acceso a internet. Ya que el ruteador estaba listo en las máquinas Debian y Mint se les configurar este nuevo router como Gateway y DNS dándoles acceso a internet.

Teniendo este pequeño grupo de máquinas funcionando se crearon los usuarios adicionales asr1-asr3, se instaló el servicio web Nginx y el servicio DNS que eran requisito para esta primera fase pero esto se realizó solo en una red por lo que se exportaron las máquinas para importarlas de nuevo la red paralela opuesta y solo fue cuestión de ajustar la dirección IP de todas las máquinas al de la nueva red en donde solamente varía la segunda posición del IP.

Fase 2 del Proyecto

Monitoreo de Redes

Se instalaron todos los elementos necesarios a cada computadora para funcionar como agentes en esta parte de monitoreo con SNMP. Para las computadoras basadas en linux se les instaló el servicio SNMPD que es el “daemon” de SNMP en donde se le asigna el Community string para que todas las computadoras o servidores tengan la misma configuración. Para el caso de Windows 8.1 este tiene una función instalada por defecto que solo se debe habilitar, así como los ruteadores pfSense en ambas redes.
Ya que todos los dispositivos fueron Agentes solo necesitábamos los administradores por lo que se instalaron 3, en Windows se instaló PRTG y en Linux se instalaron Zabbix y OpManager en donde en cada uno de ellos se le conectaba o agregaba con los agentes previamente instalados para poder acceder a su información del equipo.

Estructura del Proyecto

Instalación

El proyecto consistio en un fase de preparación y otra de instalación. Primero que nada se instalo SNMP en todos los equipos que forman parte de la red. Se hicieron las configuraciones para el monitoreo de servicios, conexiones de red de equipos y Firewalls. Y como último paso se crearon los pasos necesarios para que se envien notificaciones a correos registrados y a Telegram.
Para los 3 softwares de monitoreo utilizamos: PRTG, Zabbix y OPManager.

Zabbix

El servidor de Zabbix se utilizó exclusivamente en la red derecha o LAN 2 y fue pre-configurada para contar con todos los requerimientos y paquetes que necesita Zabbix. El único ajuste que se le tuvo que aplicar fue un cambio a la dirección IP de la misma ya que la máquina snmp2 originalmente iba a hostear el servidor de Zabbix. Una vez configurado esto solo fue cuestión de ingresar al navegador y escribir la dirección del servidor de Zabbix y se desplegó el Dashboard principal.

Antes de explicar cómo fue el procedimiento de configuración del servidor es importante recalcar que Zabbix cuenta con distintas opciones para monitorear el estado y los servicios disponibles en un equipo, una de ellas siendo SMNP. Tras una investigación, se descubrió que Zabbix cuenta con un agente que se instala en los equipos a monitorear que nos permite omitir el proceso del SMNP y simplemente conectar ambos equipos tras la instalación del agente.

Optamos por esta opción al descubrir que tanto pfSense como OPNsense también manejaban este agente. Al final nos dimos cuenta que SMNP era un poco más complicado de agregar a Zabbix que el agente, por lo que su uso nos ahorró bastante tiempo de configuración. La única desventaja fue que en vez de utilizar un community string como lo hace SMNP, Zabbix utiliza otros métodos de encriptación que resultaron un poco más complicados de implementar, por lo que decidimos no incluir algún tipo de encriptación o seguridad a la hora de que los equipos se comuniquen con el servidor.

Sin mas preambulos, continuemos a la configuración del servidor. Antes de poder ingresar, fue necesario activar el pfSense de nuestro equipo para algunas de las configuraciones que se deben cumplir. Adicionalmente, sirvió para arreglar un par de reglas que le permitieron a Zabbix monitorear a los dispositivos pfSense de la red. Para esto se instaló el agente de Zabbix en el equipo.

Por motivos de compatibilidad se recomienda adquirir la versión más reciente del agente de Zabbix aunque la documentación oficial dijera que todas las versiones a partir de la 1.8 eran compatibles con nuestra versión particular del servidor, que fue la 5. Una vez instalada se agrega una vista de configuración similar a lo que ya hemos manejado antes en pfSense, solo que esta vez dedicado la información requerida por Zabbix.

Una vez ingresada la dirección IP del servidor, no es necesario ingresar el hostname, pero para agregar una medida extra de conexión se recomienda incluirla de todas formas. Una vez especificado esto, se procedió a declarar las reglas de firewall.

Con estas reglas se asegura que la máquina pfSense pueda recibir las peticiones de Zabbix por medio del puerto 10050-10051 y que pueda devolverle la información sobre el sistema al servidor. Mientras que la conexión entre los equipos Debian de la red es de manera directa, el firewall podría parecer fácil de omitir, pero como se explicó antes Zabbix necesita estas reglas para poder monitorear el equipo por medio del agente.

Después de esto simplemente se agrega un nuevo Host en la configuración de Zabbix que coincida con la IP del equipo y para conveniencia a la hora de buscar, se encuentre bajo una categoría. Para agregar los sensores necesarios se navega a la pestaña de Templates y se eligen los sensores que se requieran.

Si la configuración es correcta, el servidor buscará que el agente se pueda comunicar devuelta, acción que toma alrededor de dos minutos. Después de esto se podrá acceder a la máquina para observar la información de cada sensor.

Para los servidores Debian, se instala el agente de Zabbix con la siguiente instrucción: apt install zabbix-agent

Si se desea confirmar que el agente esté funcionando correctamente se ejecuta la siguiente instrucción: service zabbix-agent status

Una vez instalado, es necesario realizar una pequeña modificación para que el agente pueda comunicarse con el servidor. Hay que abrir el archivo conf del agente. vim /etc/zabbix/zabbix_agentd.conf

La línea server indica cuáles direcciones podrán intercambiar mensajes por los puertos 10050 y 10051, los predeterminados de Zabbix. A este se tiene que agregar la dirección del servidor junto a la del cliente local, siguiendo la lógica de las dos reglas que se agregaron al firewall en la instalación para pfSense. Una vez editado esto, solo se agrega el host correspondiente en el servidor como se vio en los pasos de pfSense y después de unos minutos se empezará a recibir información proveniente del agente.

La configuración de OPNsense fue casi idéntica a la de pfSense, solo que la interfaz es distinta y no tuvimos acceso directo a ella. Para configurar las dos máquinas OPNsense de esta red tuvimos que comunicarnos con el equipo respectivo y guiarlos en el proceso de instalación y configuración. Tras instalar el plugin de Zabbix se procede a realizar una configuración muy similar a la de pfSense donde se declara el puerto a utilizar, las IPs que van a escuchar y las IPs que van a participar en este protocolo. Como se hizo anteriormente, al terminar de configurar esto por parte del agente, se agrega el host respectivo dentro del servidor y se espera a que la comunicación se refresque y los datos comiencen a llegar.

Para las alertas se creó un Trigger que detectara cuando un problema surgiera en alguno de los servicios pertenecientes a alguna de las máquinas que se dieron de alta en nuestro servidor.

Esta alerta es la encargada de mandar los correos electrónicos y las notificaciones de Telegram, los cuales fueron configurados aparte. La configuración del correo electrónico se llevó a cabo por medio de Media Types, una función de Zabbix que permite configurar rápidamente distintos servicios de correo electronico o mensajeria con el propósito de generar alarmas. Para la configuración de correo electrónico basto con especificar un servidor SMTP y la cuenta perteneciente a dicho servidor.

Para que la acción de Trigger pudiera mandar exitosamente el correo electrónico fue necesario configurar el usuario administrador para que manejara las distintas cuentas de correo de todos los integrantes del equipo.

Telegram requirió un poco más de configuracion externa. Primero se creó un bot de Discord que se encargara de recibir estos mensajes y mandarlos a un grupo de chat mediante un webhook.

Devuelta en el servidor de Zabbix, se utilizó el token del bot para que este recibiera los mensajes de las alarmas. Similar al correo electronico, se modificó al usuario administrador para que el destino de los mensajes fuera el grupo de Telegram creado anteriormente.

Y con esto se concluyo la configuración de nuestro servidor Zabbix en la red derecha.

PRTG

Este Software de monitoreo fue instalado en el sistema operativo Windows 8.1. La configuración inicial es agregar el "community string" asignado para que al agregar las computadoras puedan validar la comunicación.

De esta manera es posible agregar los dispositivos para comenzar a ser analizados. Estos dispositivos se agregan seleccionando el menú “devices” en la parte superior de la interfaz seguido del botón de "más".

Después se escribe un nombre como identificador y el IP necesario para realizar la conexión

Una vez agregado un nuevo dispositivo, podemos agregarle sensores que se comuniquen con el agente del dispositivo para que recuperen la información solicitada. Para agregar un sensor se hace sobre el mismo menú de agregar dispositivos, pero en el menú posterior debes seleccionar el dispositivo al que se le quiera agregar el sensor.

Al Seleccionar el equipo, nos trasladara a la sección en donde puedes seleccionar el sensor dependiendo de las necesidades del dispositivo, ya sea de desempeño o de servicios.

Por defecto se mandan alertas cuando un sensor no está dando los resultados esperados. Estas alertas intentan mandar un correo a los administradores del sistema, pero no hay datos por lo que se tienen que agregar bajo el menú “Setup->System Administration->Notification Delivery” en donde ponemos la información del servidor de correo. Para casos de prueba se utilizo Gmail.

Se creó un segundo método para entregar las alertas cuando algo no está correcto en la red por medio de la plataforma Telegram. Primero se creó un bot en blanco para utilizar como base. Este se agrego al grupo de notificacion deseado y teniendo el URL del bot así como el id del chat al que le mandara las notificaciones, podemos ir al menú “Setup->Account Settings->Notification Templates” en el cual agregamos un nuevo template donde enlazaremos el bot de telegram.

Se debe encender la opción de “Execute HTTP Action” y agregar el URL y el código que interpreta telegram para el envío de notificaciones.

Ahora para ponerlo a funcionar se necesita acceder al menú “Devices” dentro seleccionar “Notification Triggers” y agregar una nueva regla igual a por defecto pero realizando la acción recién agregada.

Para crear alertas o alarmas personalizadas dentro de la misma aplicación se realiza de la misma manera que como el método de telegram pero habilitando la sección “Add Entry to Event Log” y seleccionando lo que se necesite ya sea una alerta con“Error”, una alarma con “Warning” o mostrar información con “Information”.
